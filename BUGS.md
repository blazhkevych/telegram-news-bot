### БАГ-010 — bot.py обрізаний: main() ніколи не викликався (канал мовчав)
- **Дата:** 2026-07-14. **Помітив:** Claude (аналіз коду за запитом власника).
- **Симптом:** з коміту `a74df05` (сесія 7) бот не публікував НІЧОГО — жодних
  постів, жодних помилок у логах чи підсумку адміну. Виглядало як «все ок,
  просто немає новин».
- **Причина:** той коміт (правки RSS_FEEDS) випадково обрізав хвіст `bot.py` —
  зник блок `notify_admin(summary)` + `conn.close()` + `if __name__ == "__main__":
  main()`. Файл лишався синтаксично валідним Python (обрізаний рядок був просто
  «висячим» виразом), тому `python bot.py` не падав з винятком — модуль просто
  визначав функції й завершувався, ніколи не викликаючи `main()`. Той самий клас
  інциденту, що вже описаний у сесії 7 як «Windows-маунт обрізає файли на
  кирилиці», тільки цього разу потрапило в git і пішло на прод.
- **Рішення:** відновлено видалений хвіст; заразом виправлено ще 2 суміжні баги
  в тому ж файлі, знайдені при читанні коду: `is_russian(title, title)` →
  `is_russian(title, summary)` (перевірка мови ігнорувала текст новини) і
  прибрано мертвий gate «чекати підтвердження теми» в `main()` (лічильник теми
  вже інкрементувався в `fetch_news()` до цієї перевірки, тож умова майже
  ніколи не спрацьовувала за задумом).
- **Верифіковано:** `ast.parse` на bot.py OK; `grep "__main__"` підтверджує
  рівно один виклик `main()`; закомічено й запушено (`1648278`).
- **Статус:** ✅ 2026-07-14.

### БАГ-008 — Gemini 404 (мертва модель) + оманливе обрізання логу
- **Дата:** 2026-07-13. **Помітив:** власник (фідбек-бот) + Claude (читання коду).
- **Симптом:** у підсумку адміну — `Gemini: 404 Client Error: Not Found for url:
  https://generativelanguage.google`. Виглядало як поламаний/обрізаний домен.
- **Причина (двошарова):**
  1. URL у `bot.py` НЕ зламаний — повний і правильний. «...generativelanguage.google» —
     це рівно перші 70 символів тексту помилки, бо лог робив `str(e)[:70]`.
  2. Модель `gemini-2.5-flash` Є у ListModels (HTTP 200), але при генерації дає
     `404 NOT_FOUND: "no longer available to new users"` — Google закрив її для
     нових користувачів. Клас БАГ-001 (Cerebras 404 — модель недоступна акаунту).
- **Рішення:**
  • `call_llm`: додано гілку `status >= 400`, що логує тіло відповіді (`r.text`,
    ≤300 симв. консоль / ≤120 підсумок); обрізання в `except` 70→120. Саме це
    й показало точну причину.
  • `gemini_check.py`: перебирає кандидатів через РЕАЛЬНИЙ виклик генерації
    (присутність у ListModels ≠ доступність) і класифікує 200/404/503/429.
  • Модель: `bot.py` `gemini-2.5-flash` → **`gemini-3.5-flash`**. Діагностика
    власником підтвердила: 3.5-flash віддає 503 (не 404!) — тобто НАЗВА ВАЛІДНА,
    просто транзієнтне «high demand». `2.5-flash-lite` → 404 (теж закрита),
    `2.0-flash` → 429 (квота). Gemini — резервний провайдер (останній у ланцюзі),
    тож епізодичні 503/429 безпечні: `call_llm` просто не перепише цей пост і
    наступний запуск (щогодини) спробує знову. Ідемпотентно.
